Section 01

Executive Summary

M5 defines the operational layer that governs live validator nodes once they are registered and active on the Post Fiat Superstructure. Where M3 established the static architecture and M4 validated readiness, M5 specifies the continuous runtime systems that monitor node health, detect anomalies, coordinate fleet management, and execute graduated response protocols without requiring human intervention for routine degradation events.

The M5 architecture is organized around a single operational principle: every response to node degradation must be proportional, reversible, and auditable. The Orchestration Control Plane does not treat all failures equally. A transient latency spike and a sustained telemetry blackout require different responses — and this specification defines both the thresholds that distinguish them and the escalation paths that govern what happens next.

This document covers four operational domains: automated health scoring with sustain windows to prevent false-positive failovers; telemetry anomaly detection with typed anomaly classifications; the OCP fleet management interface and its data visualization requirements; and the full escalation ladder from initial degradation detection through soft-failover to hard failover, including recovery paths at each stage.

Architecture Continuity This specification is a downstream implementation of the Superstructure Validator Routing Logic (M3) and the Failover and Routing Stress Test (M4). All Composite Routing Score weights, failover SLAs, signature scheme parameters, and zero-custody constraints established in prior phases remain in force. This document extends those foundations into live operational procedure.
Section 02

Automated Health Scoring Thresholds and Telemetry Anomaly Detection

2.1 Composite Routing Score Architecture

Every active operator node carries a Composite Routing Score (CRS) that the OCP recalculates every 60 seconds from four telemetry inputs. The CRS is the primary operational signal — routing weight, escalation state, and failover decisions all derive from it.

CRS Formula
CRS = (VI_score × 0.40) + (uptime_24h × 0.30) + (latency_score × 0.20) + (telemetry_freshness × 0.10)
Component Derivations:
VI_score = Verified Inference reliability index [0.00–1.00]
uptime_24h = (slots_attested / slots_eligible) over rolling 24h window
latency_score = max(0, 1 - (p95_latency_ms / 400)) — floored at 0 above 400ms
telemetry_fresh = 1.00 if last beacon < 30s ago; 0.00 if beacon gap > 90s

CRS outputs a continuous value between 0.00 and 1.00. The OCP operates on normalized CRS but threshold triggers are expressed as percentages (CRS × 100) throughout this document for operational readability.

2.2 Health State Zones

The CRS range is divided into four non-overlapping health state zones. Zone assignment determines which monitoring mode and response protocol applies to the node.

NOMINAL
≥ 70
Full routing weight. Standard monitoring interval. No operator alert.
WATCH
60–69
Reduced weight by 30%. Operator alert triggered. Sustain window active.
DEGRADED
40–59
Weight reduced 60%. Soft-failover preparation begins. Escalation timer active.
CRITICAL
< 40
Immediate routing exclusion. Hard-failover authorized. Recovery requires ceremony.
Hysteresis Requirement Transitions between adjacent zones require the CRS to sustain the new zone value for at least two consecutive 60-second scoring cycles before the zone assignment updates. This prevents oscillation between zones during transient fluctuations. A single below-threshold score does not change zone assignment.

2.3 Sustain Windows and False-Positive Prevention

Each alert condition has a sustain window — a minimum duration during which the condition must persist before the OCP acts on it. Sustain windows exist specifically to prevent false-positive escalations triggered by transient network events, momentary latency spikes, or brief telemetry gaps that resolve without intervention.

Condition Raw Trigger Value Sustain Window Poll Count Required Action on Confirm
CRS enters Watch zone CRS 60–69 120 seconds 2 of 2 cycles WATCH state, -30% weight, operator alert
CRS enters Degraded zone CRS 40–59 120 seconds 2 of 2 cycles DEGRADED state, -60% weight, soft-failover prep
CRS enters Critical zone CRS < 40 60 seconds 1 of 1 cycles CRITICAL state, full routing exclusion
p95 latency elevated > 400ms 60 seconds 1 of 1 cycles Latency anomaly flag, CRS penalty applied
Telemetry beacon gap > 30s no beacon 60 seconds 1 of 1 cycles Telemetry anomaly, freshness score drops to 0.50
Telemetry blackout > 90s no beacon 0 seconds Immediate Telemetry score to 0.00, BLACKOUT anomaly triggered
VI score degraded VI < 0.60 120 seconds 2 of 2 cycles VI anomaly flag, Degraded zone entry if CRS follows
maintenanceDeclared flag Operator-set 0 seconds Immediate Routing weight zeroed, MAINTENANCE mode, no CRS penalty

The sustain window mechanism works as follows: when a threshold condition is first observed, the OCP creates a pending alert record with a timestamp. It does not act on the condition. At the next scoring cycle, if the condition still holds, the sustain window logic checks elapsed time. If the sustain duration has been met, the alert is confirmed and the associated action fires. If the condition has cleared during the window, the pending alert is discarded and no action is taken.

Three-Poll Buffer Rationale (Critical Zone Exception) Critical zone entry requires only a single confirmed cycle because the CRS value below 40 represents a compound failure: multiple scoring inputs have degraded simultaneously. At that failure depth, sustain windows provide false assurance rather than false-positive protection. The 60-second window at critical allows a single scoring cycle to complete without creating a meaningful delay in protective response. Watch and Degraded zones use two-cycle confirmation because individual component fluctuations can push CRS below 70 or 60 transiently without indicating systemic degradation.

2.4 Telemetry Anomaly Detection

The OCP classifies telemetry anomalies into five typed categories. Each type has a detection mechanism, a confirmation threshold, and a named anomaly identifier that appears in operator alerts and the audit log.

Anomaly Type Identifier Detection Mechanism Confirmation Impact
Telemetry Gap ANOM_TEL_GAP Beacon interval > 30s 60s sustained Freshness score 0.50, watch flag
Telemetry Blackout ANOM_TEL_BLACKOUT Beacon interval > 90s Immediate on detection Freshness score 0.00, escalation trigger
Latency Spike ANOM_LAT_SPIKE p95 latency > 400ms 60s sustained Latency score floors to 0.00
VI Degradation ANOM_VI_DEGRADE VI score < 0.60 120s sustained CRS penalty, escalation if compound
Slot Attestation Miss ANOM_SLOT_MISS 3+ consecutive missed slots Immediate on third miss Uptime score revision, watch flag

Compound anomalies — where two or more typed anomalies are active simultaneously — trigger an automatic escalation path review regardless of the current CRS zone. If ANOM_TEL_BLACKOUT coincides with ANOM_VI_DEGRADE, the OCP treats this as a Critical-priority event and bypasses Watch and Degraded zone transitions entirely, proceeding directly to Critical state assessment.

2.5 Anomaly Decay and Auto-Resolution

Anomaly flags are not permanent. Each anomaly has an auto-resolution condition: when the underlying metric returns to nominal range and remains there for a defined resolution window, the anomaly flag clears automatically and the CRS recalculates using current inputs. The node may transition back through Degraded and Watch zones before returning to Nominal, following the same two-cycle sustain requirement in reverse.

Anomaly Resolution Condition Resolution Window
ANOM_TEL_GAP Beacon interval returns to < 20s 60s sustained
ANOM_TEL_BLACKOUT Beacon received and subsequent gap < 30s 120s sustained
ANOM_LAT_SPIKE p95 latency < 300ms 120s sustained
ANOM_VI_DEGRADE VI score returns to > 0.70 120s sustained
ANOM_SLOT_MISS 10 consecutive successful attestations N/A (count-based)
Section 03

Orchestration Control Plane Fleet Management Interface

3.1 OCP Interface Architecture

The Orchestration Control Plane Fleet Management Interface (OCP-FMI) is the operator-facing surface through which fleet health, routing weights, anomaly states, and escalation status are monitored and controlled. The OCP-FMI is a read-heavy interface: operators observe state and acknowledge alerts, but the OCP executes routing changes autonomously. Direct operator override of routing weights is restricted to MAINTENANCE mode declarations.

Primary Display Panels
  • Fleet Health Overview — aggregate CRS heatmap, state zone distribution
  • Node Detail View — per-node CRS, component scores, active anomalies
  • Routing Weight Distribution — live weight allocation across active nodes
  • Escalation Queue — nodes in Watch/Degraded/Critical with timer status
  • Anomaly Event Log — typed anomaly history with timestamps and resolution status
  • Signing Ceremony Queue — pending ceremony actions required for node recovery
Operator Control Actions
  • Declare Maintenance Mode — zero routing weight, no CRS penalty, signed ceremony required to exit
  • Acknowledge Alert — log operator awareness, does not alter OCP behavior
  • Request Manual Failover Review — elevates node to permanent removal consideration (MANUAL path)
  • View Escalation History — full audit trail for any node since activation
  • Export Telemetry Report — operator-scoped data export for external review

3.2 Data Visualization Requirements

The OCP-FMI must present fleet data in a way that allows operators to distinguish between momentary anomalies and sustained degradation at a glance. The following visualization requirements define the minimum display standard for the interface.

Fleet Health Heatmap

  • Each node represented as a tile, color-coded by health zone: green (Nominal), amber (Watch), orange (Degraded), red (Critical), grey (Maintenance)
  • Tile brightness scales with CRS value within zone — a CRS of 68 renders differently from a CRS of 61 within the Watch band
  • Tiles pulse with a slow animation when a sustain window is active but not yet confirmed, indicating pending state change
  • Click on any tile expands to Node Detail View

CRS Time-Series Chart

  • Per-node view showing CRS value over rolling 24-hour window, updated every 60 seconds
  • Horizontal zone threshold lines rendered at CRS values 70, 60, and 40
  • Anomaly events plotted as vertical markers on the time axis with anomaly identifier tooltip
  • Routing weight overlaid as a secondary shaded area beneath the CRS line
  • Failover events marked with a distinct icon and linked to the escalation log entry

Escalation Queue Panel

  • Sorted by urgency: Critical nodes first, then Degraded (sorted by time-in-zone descending), then Watch
  • Each row shows: node ID, current CRS, health zone, time in current zone, active anomaly codes, next escalation action and ETA
  • Countdowns rendered in seconds for Critical nodes, minutes for Degraded and Watch
  • Acknowledged alerts shown with operator ID and acknowledgment timestamp

Routing Weight Distribution

  • Real-time donut or stacked bar showing current routing weight share across all ACTIVE operators
  • Maximum single-operator weight ceiling of 60% rendered as a hard limit line — auto-visualization makes concentration risk visible
  • Historical weight distribution overlay available for comparison against current state

3.3 OCP Scoring Cycle Mechanics

The OCP runs a deterministic 60-second scoring cycle. The cycle sequence is fixed so alert evaluation and routing updates cannot race against each other within a cycle.

Cycle T+0:   Collect telemetry inputs from all active nodes
Cycle T+5s:  Compute component scores (VI, uptime, latency, freshness)
Cycle T+10s: Compute CRS for each node
Cycle T+15s: Evaluate zone transitions against sustain window history
Cycle T+20s: Confirm or discard pending alerts
Cycle T+25s: Execute routing weight updates for confirmed state changes
Cycle T+30s: Broadcast weight update to active middleware instances (SLA: within 30s of trigger)
Cycle T+35s: Write cycle results to audit log
Cycle T+60s: Next cycle begins
Atomic Weight Update Routing weight redistribution at T+25s is executed as a single atomic operation across all nodes in the affected group. Partial updates — where some nodes receive the new weight before others — are rejected. If the atomic write fails, the OCP retains the previous weight distribution and flags the cycle as a write-error event in the audit log. The next cycle will retry the update.

3.4 Fleet Minimum Viability Constraint

The OCP enforces a hard floor of two active operators at all times, consistent with the minimum set in prior phases. If a failover event would reduce the active operator count below two, the OCP halts the failover execution, flags the event as ERR_FLEET_MIN_VIOLATION, and escalates immediately to a MANUAL review path requiring human operator decision. The failing node is placed in a zero-weight holding state rather than full deregistration until the human review resolves.

Section 04

Escalation Paths and Soft-Failover Architecture

4.1 Node Operational State Machine

Every node moves through a defined set of operational states. State transitions are governed by CRS thresholds, sustain window confirmations, and operator declarations. The state machine has no cycles that can trap a node indefinitely — every degraded state has a defined recovery path.

ACTIVE
WATCH
SOFT_FAILOVER
HARD_FAILOVER

Additional states: MAINTENANCE (operator-declared, reversible without ceremony), RECOVERY (node has exited hard failover and is rebuilding score), and DEREGISTERED (permanent removal, requires signing ceremony to re-enter).

4.2 Full Escalation Path Specification

1
INITIAL DETECTION — WATCH STATE ENTRY

CRS falls into Watch zone (60–69) and sustains for two consecutive 60-second cycles (120 seconds total). OCP confirms zone transition, reduces routing weight by 30%, and emits ALERT_WATCH_ENTRY to the Operator Notification Bus. The node remains in active service at reduced weight. No signing ceremony is required or triggered. The operator is expected to acknowledge the alert within 15 minutes — acknowledgment is logged but does not change OCP behavior.

2
ESCALATION — DEGRADED STATE ENTRY

If CRS further declines into the Degraded zone (40–59) and sustains for two cycles, the OCP transitions the node to DEGRADED state. Routing weight is reduced by 60% from nominal. A Soft-Failover Preparation flag is set, meaning the OCP begins pre-positioning load redistribution across remaining ACTIVE operators so that if a soft-failover fires, the weight reassignment can complete within the 30-second SLA. The operator receives ALERT_DEGRADED_ENTRY with a 10-minute escalation timer visible in the OCP-FMI escalation queue.

3
SOFT FAILOVER — LOAD REDISTRIBUTION WITHOUT DEREGISTRATION

If a node in DEGRADED state has not recovered to Watch or Nominal after three consecutive 60-second cycles in Degraded (180 seconds of sustained degradation), the OCP executes a Soft Failover. In Soft Failover, the node's routing weight is set to zero and all traffic is redistributed to remaining ACTIVE operators. Critically, the node is NOT deregistered. It is placed in a SOFT_FAILOVER holding state and continues to receive telemetry monitoring. It remains eligible for automatic recovery: if the CRS recovers above 60 and sustains for two cycles, the node re-enters the ACTIVE routing pool with a graduated weight ramp — 20% initial weight, increasing by 10% per cycle as CRS remains above 70, up to the pre-failover allocation ceiling.

4
HARD FAILOVER — ROUTING EXCLUSION AND DEREGISTRATION

Hard failover fires when one or more of the following conditions occur: CRS falls below 40 and sustains for one 60-second cycle; telemetry blackout (ANOM_TEL_BLACKOUT) persists for 90 seconds; compound anomalies of types ANOM_TEL_BLACKOUT and ANOM_VI_DEGRADE are simultaneously active; or ERR_FLEET_MIN_VIOLATION is resolved in favor of removal via MANUAL review. Hard failover is an AUTO-authorized action — no operator confirmation is required before execution. The node is immediately removed from the routing pool, its weight redistributed atomically, and an ERR_HARD_FAILOVER event is written to the audit log. The failover SLA from trigger detection to weight update broadcast is 30 seconds. Recovery from hard failover requires a signing ceremony to re-register the node on the network.

4.3 Soft Failover vs. Hard Failover Decision Matrix

Condition CRS Range Duration Required Action Ceremony Required Authorization

Degraded zone entry 40–59 120s sustained -60% weight, prep No AUTO
Sustained Degraded (no recovery) 40–59 180s in Degraded Soft Failover (zero weight, node held) No AUTO
Critical zone entry < 40 60s (1 cycle) Hard Failover Yes (to recover) AUTO
Telemetry Blackout Any 90s no beacon Hard Failover Yes (to recover) AUTO

4.4 Recovery Paths

Recovery paths vary by the state a node exited from. Nodes that self-recovered from Watch or Degraded re-enter with no weight penalty and no ceremony requirement. Nodes that were soft-failed over re-enter via the graduated ramp described in Section 4.2. Nodes that were hard-failed over require a full signing ceremony to re-register, after which they enter RECOVERY state and are treated as new nodes for routing weight allocation purposes, starting at the minimum allocation floor.

Zero-Custody Constraint on Recovery The signing ceremony required for hard-failover recovery follows the same four-step process established in M3: envelope delivery to institution, mandatory human verification inside custody, signing inside custody, broadcast. The private key does not cross the custody boundary at any point. The recovery ceremony is initiated by middleware constructing the unsigned re-registration payload — not by the operator. This preserves zero-custody architecture through the recovery workflow as well as the initial activation workflow.

4.5 Escalation Notification Obligations

The Operator Notification Bus must deliver escalation alerts within 60 seconds of the OCP confirming the triggering condition. Delivery failure (no acknowledgment from the Notification Bus within 30 seconds of dispatch) triggers a retry with exponential backoff: 30s, 60s, 120s. After three failed delivery attempts, the OCP logs an ERR_ALERT_DELIVERY_FAIL event but continues to execute routing changes autonomously. Escalation actions do not wait for alert delivery confirmation.

Alert Code Trigger Condition Operator Required Response Response Window
ALERT_WATCH_ENTRY CRS enters Watch zone Acknowledge; investigate root cause 15 minutes
ALERT_DEGRADED_ENTRY CRS enters Degraded zone Acknowledge; begin remediation 10 minutes
ALERT_SOFT_FAILOVER Soft-failover executed Acknowledge; document cause; remediate 30 minutes
ALERT_HARD_FAILOVER Hard-failover executed Acknowledge; initiate recovery procedure; report to network 60 minutes
ALERT_FLEET_MIN Fleet minimum risk detected Participate in MANUAL review decision 30 minutes
Section 05

Operational Integration with M3/M4 Infrastructure

5.1 Continuity with Prior Phase Specifications

M5 Live Operations does not introduce new cryptographic primitives, custody models, or routing architectures. It operates within the constraints established across M2 through M4 and extends them into runtime behavior. The following established parameters govern M5 operations without modification.

  • Signature scheme: Ed25519 over canonical JSON (RFC 8785) — applies to all signing ceremonies including recovery ceremonies
  • Ceremony envelope freshness window: 120 seconds — applies to recovery ceremony envelopes constructed by middleware
  • Nonce retention period: 30 days minimum — applies to recovery ceremony nonces as with initial registration
  • Replay protection error: ERR_NONCE_REPLAY — fired on duplicate nonce in any ceremony type
  • Role model: Maker / Checker / Reviewer — applies to recovery ceremony authorization inside institution custody
  • Maximum single-operator routing weight: 60% — enforced by OCP weight distribution logic, visible in OCP-FMI
  • Minimum active operators: 2 — enforced by ERR_FLEET_MIN_VIOLATION gate before any hard-failover execution
  • Failover SLA: 30 seconds from confirmed trigger to weight update broadcast — applies to both soft and hard failover events

5.2 Verified Inference Integration

Verified Inference (VI) scores are produced by the Post Fiat network's inference reliability system and are consumed by the OCP as one of four CRS inputs. M5 operations treat VI scores as an external input signal — the OCP does not modify or override VI scores. When VI scores indicate degradation (ANOM_VI_DEGRADE), the OCP responds through the standard anomaly and escalation framework without contacting or adjusting the VI system directly.

The relationship between VI score degradation and network participation probability is governed by the Verified Inference model, not by the OCP. The OCP's role is to translate VI score trends into routing weight adjustments and escalation decisions that protect the network from relying heavily on a node whose inference reliability is declining before that decline reaches a hard threshold in the VI system itself.

5.3 Audit Log Requirements

Every OCP action — scoring cycle result, state transition, alert dispatch, weight update, failover execution, recovery event — must be written to an immutable audit log accessible to network reviewers. The audit log is append-only and timestamped to millisecond precision. Log entries include the triggering condition, the sustain window evaluation result, the action taken, and any error codes generated.

Network Review Access The audit log is readable by authorized network reviewers without exposing private key material, operator authentication credentials, or institution-internal signing ceremony records. The log contains operational metadata only: node IDs, CRS values, timestamps, action codes, and anomaly identifiers.

Appendix B — OCP State Machine Transitions
State: ACTIVE
  → WATCH          if CRS < 70 for 2 cycles (120s)
  → MAINTENANCE    if maintenanceDeclared = true (immediate)
  → CRITICAL       if CRS < 40 for 1 cycle (60s) — bypasses Watch/Degraded

State: WATCH
  → ACTIVE         if CRS ≥ 70 for 2 cycles (auto-recovery)
  → DEGRADED       if CRS < 60 for 2 cycles (120s)
  → MAINTENANCE    if maintenanceDeclared = true

State: DEGRADED
  → WATCH          if CRS 60–69 for 2 cycles (partial recovery)
  → ACTIVE         if CRS ≥ 70 for 2 cycles (full recovery)
  → SOFT_FAILOVER  if 3 consecutive cycles in DEGRADED (180s) with no recovery
  → HARD_FAILOVER  if ANOM_TEL_BLACKOUT + ANOM_VI_DEGRADE compound (immediate)

State: SOFT_FAILOVER
  → ACTIVE(ramp)   if CRS ≥ 60 for 2 cycles — graduated weight ramp
  → HARD_FAILOVER  if CRS < 40 for 1 cycle while in SOFT_FAILOVER

State: HARD_FAILOVER
  → RECOVERY       after signing ceremony completes (operator re-registration)

State: RECOVERY
  → ACTIVE         after 3 consecutive cycles with CRS ≥ 70

State: MAINTENANCE
  → ACTIVE         after operator clears maintenance flag + signing ceremony exit

State: DEREGISTERED
  → (terminal)     permanent removal, requires full onboarding to return
